S-01 Registration and Landing Screen

The registration screen is the first thing a new user sees. The left panel establishes the value proposition in three proof points. The right panel is a minimal registration form. Google OAuth is offered as the primary path; email/password is secondary. The screen must load in under 1.5 seconds; no animation or asset that delays this.

Implementation Notes

1
Google OAuth button is styled as secondary (ghost button) but positioned first. Research shows conversion is higher when OAuth is the first option but not the most visually dominant, reducing pressure for users who prefer email.
2
Left panel is hidden on mobile (viewport under 768px). The value proposition moves to a 3-line summary above the form on mobile, not a separate panel.
3
No password strength meter on registration — it adds visual noise and the bcrypt requirement is minimum 8 characters. The error state appears only after submission attempt with a too-short password.
4
On successful registration, the page does NOT redirect to a dashboard. It advances to S-02 (Profiling Q1) without a page reload, using a slide-in animation on the right panel only.

Implementation Notes

1
High-risk flag (Q5) appears via JavaScript as the user types, triggered by keyword matching against the detection list (promise, refund, delete, medical, financial, guarantee, irreversible). The flag disappears if the high-risk words are removed. The flag text references exactly what will be added so there are no surprises on the confirmation screen.
2
Q4 checkboxes use tag-style pills rather than a grid to conserve vertical space. Selecting 'Customer messages' triggers a visual note below the tags: "We'll add a privacy rule to protect customer personal information."
3
Q6 plain-English options map to governance modes in the backend (not very = light, check what it did = standard, prove to someone = audited) but this mapping is NEVER shown on this screen. It is internal only.
4
Placeholder text in Q2 is dynamically set based on the use case routing from Q1: a user who selected 'Build something new' sees a customer support example; a user who selected 'Fix sessions that lose context' sees a development session example.

Implementation Notes

1
Invariant checkboxes are checked by default. Unchecking a box marks that invariant as excluded from the scaffold; it is not created. The user is not warned or asked to confirm unchecking — their choice is respected without friction.
2
The amber-highlighted invariant (Q5-triggered human confirmation gate) is visually distinguished because it was triggered by the user's own words, not by the category template. The "(Added because of your answer to Q5)" note makes the origin transparent.
3
Bootstrap prompt preview is truncated with a "click to preview" affordance. The full prompt opens in a modal. This prevents the confirmation screen from being overwhelming while giving curious users access to the full content.
4
On "This Looks Right": all records are created in a single transaction; the user sees a 300ms spinner; then they land on S-07 (first session dashboard). No intermediate "success" screen.

Implementation Notes

1
Sidebar vocabulary in Guided mode: "Activity History" (not "Decision Log"), "Connected Apps" (not "Connectors"), "Usage" (not "Token Tracking"). The Novice vocabulary table from WP-012 Section 8 drives all sidebar labels.
2
Copy button behavior: on click, the button text changes to "✓ Copied" for 2 seconds, then reverts. No modal, no download, no new window — clipboard only. If the clipboard API is unavailable (some mobile browsers), a modal appears with the full prompt in a select-all textarea.
3
"Log This Response" is not shown on this screen — it appears only after the user has clicked "Copy Starter" at least once. This prevents first-time confusion from a field they do not yet understand they need. After the copy click, the "Log This Response" textarea slides in below the copy box with a label: "Paste your AI's response here to log it."
4
Stats grid in Guided mode shows plain-English sublabels under every number. In Standard mode these sublabels are hidden (assumed understood). In Expert mode the stat cards show the database field name on hover.

Implementation Notes

1
The wow moment card replaces the empty state in the decision log section — it does not appear as a separate modal or overlay. The transition is a fade-in: the empty state fades out (150ms) and the wow card fades in (300ms).
2
Tooltip on each stat appears on hover (desktop) or tap (mobile). The tooltip is the plain-English explanation of the number. In Standard mode, tooltips are still present but shorter. In Expert mode, the tooltip shows the database field name and table instead.
3
wow_moment_reached flag is set to true in user_onboarding as a background API call when this card renders. This triggers the abandonment email suppression (no re-engagement email if wow moment was reached) and starts the 48-hour countdown for the second-session prompt.

S-13 Onboarding State Machine — All States and Transitions

The complete state machine for the user_onboarding.current_step field. Each state has a defined entry condition, exit condition, and timeout behavior.

STATE: profiling
User has registered but not completed the 3 profiling questions. Dashboard is blocked — any attempt to navigate to the main dashboard redirects to the profiling flow.
Entry: User_onboarding row created on registration. Exit: All 3 profiling questions answered and Continue clicked on Q3. Timeout: After 24 hours in this state, send re-engagement email with a "Continue your setup" CTA.
STATE: bootstrap_interview
Profiling complete. User is on the 6-question bootstrap interview. Dashboard still blocked. The Back button on the interview returns to profiling Q3 (not Q1).
Entry: Q3 submitted; routing from Q1 is "Build something new." Other Q1 options skip this state. Exit: All 6 interview questions answered and "Build My Project" clicked. Timeout: After 24 hours, send re-engagement email with the interview answers pre-populated and a "Finish setting up" CTA. Include the answers they already gave so they do not need to re-enter them.
STATE: model_setup
Bootstrap interview complete; scaffold generated. User is on the scaffold confirmation screen. This is the only step where the user can modify what was generated before it is saved.
Entry: Scaffold generator completes successfully. Exit: User clicks "This Looks Right" and the transaction commits. Timeout: After 48 hours in this state (most common abandonment point), send email with the scaffold summary and a "Your project setup is waiting" CTA; include a direct link that resumes from the confirmation screen.
STATE: first_session
Scaffold saved. User has landed on the dashboard for the first time. The first-session banner is visible. Dashboard is fully accessible in Guided mode. wow_moment_reached is false.
Entry: Scaffold confirmation transaction commits. Exit: wow_moment_reached = true (first decision is logged). Timeout: After 48 hours without a logged decision, send email with the bootstrap prompt embedded directly in the email body and a single CTA: "Paste this into Claude or ChatGPT to get started."
STATE: complete
Onboarding finished. wow_moment_reached = true. Dashboard shows full Guided mode UI. Second-session prompt will appear on next login within 48 hours. All re-engagement emails suppressed.
Entry: First decision logged; wow_moment_reached set to true. Exit: This is a terminal state — no exit. Post-completion: onboarding_completed_at timestamp set; Day 7 retention measurement starts.
S-14 Progressive Disclosure Settings Toggle

The Settings screen where users can change their UI mode at any time. The current mode is highlighted. Changing mode takes effect immediately on the current page without a reload.

contextkeeper.org/app/settings S-14

Interface Mode

Choose how technical the platform's language should be. This affects labels, tooltips, and which advanced settings are visible. Your data and capabilities are the same at every level.

Guided
for new users
Plain English throughout. "Rules" instead of "Invariants." Step-by-step guidance on first use of every feature.
●
Standard
current
Technical terms with brief definitions on first use. Full feature set visible. Suitable for developers and engineers.
Expert
for engineers
Full technical vocabulary. Database field names on hover. Raw governance_config JSON editor. Links to white paper sections for every concept.
Changes take effect immediately. No page reload required.

Implementation Notes

1
Mode change is instant via a JavaScript event that updates a data-ui-mode attribute on the document body and triggers a CSS class swap. All label elements have data-novice, data-standard, and data-expert attributes; the active mode determines which is displayed. No page reload.
2
The "current" badge appears next to the active mode. When the user selects a different mode, the badge moves immediately on selection (before save) to provide immediate visual feedback that the change was registered.
3
Architect / Evaluation mode is not listed here — it is only available to Enterprise accounts and is activated automatically for the Enterprise onboarding path. It can be deactivated by setting to Expert mode from this screen.
