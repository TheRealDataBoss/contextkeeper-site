The Elevator Pitch

What contextkeeper does, in plain language

Imagine you're building a house with 3 different contractors who never talk to each other. Every time one leaves, the next one has to figure out what was done by walking around the house. contextkeeper is the clipboard that records every decision, every measurement, every change - and hands it to the next contractor so they can start working in 60 seconds.

Why this matters

Saves hours every week

Developers spend 15-30 minutes re-explaining their project every time they start a new AI chat. contextkeeper eliminates that completely.

Prevents costly mistakes

When a new AI session doesn't know about prior decisions, it contradicts them. contextkeeper ensures every session respects what came before.

Works with any AI

Not locked to one vendor. Use Claude today, switch to GPT tomorrow, try Gemini next week. contextkeeper works with all of them.

The Quickstart

Up and running in 5 commands

pip install contextkeeper
terminal - quickstart
# 1. Initialize in your project
$ contextkeeper init
Initialized contextkeeper in ./my-project
Created STATE_VECTOR.json + .contextkeeper/
 
# 2. Sync after an AI session
$ contextkeeper sync --agent claude
Captured 14 decisions, 3 invariants, 47 files
Session state saved. SHA256: a3f8c2...
 
# 3. Check project state
$ contextkeeper status
Project: my-app
Sessions: 14 | Agents: 3 | Decisions: 89
Last sync: 2 minutes ago (claude)
 
# 4. Bootstrap a new AI session
$ contextkeeper bootstrap --agent gpt-4
Loading 14 sessions into GPT-4...
0 drift. Ready to continue.
 
# 5. Health check
$ contextkeeper doctor
All checks passed (6/6)
STATE_VECTOR: valid | Config: valid | SHA: verified

What just happened

init

Created STATE_VECTOR.json (the core state file) and the .contextkeeper/ config directory in your project root.

sync

Captured your current AI session state: decisions made, files touched, invariants established, and tagged it with the agent name.

bootstrap

Loaded the full project state into a new AI session. The new agent knows every decision, every file, every constraint.

Python SDK usage

session_example.py 
from contextkeeper import Session, Decision

# Create a session
session = Session.create("my-project", agent="claude")

# Record a decision
session.add_decision(Decision(
    title="Use Redux Toolkit for state management",
    rationale="Simpler than raw Redux, built-in immer",
    alternatives_rejected=["Zustand", "MobX", "Jotai"]
))

# Sync state
session.sync()

# Later: bootstrap into a new agent
state = Session.bootstrap("my-project", agent="gpt-4")
print(state.decisions)  # All decisions from all sessions

Where files live

my-project/
  STATE_VECTOR.json - core project state
  .contextkeeper/
    config.json - project settings
    sessions/
      001-claude.json - session snapshots
      002-gpt4.json
    bundles/
      handoff-v3.zip - portable bundle
    decision_log.json - all decisions
The Architecture

DEL-v2 governance protocol deep dive

The formal state machine, dual-SHA integrity, invariant registry, and multi-agent coordination model.

DEL-v2 state machine (10 states)

S0 UNINITIATED
S1 INITIATED
S2 PLANNING
S3 EXECUTING
S4 GATE_CHECK
S5 AWAITING_MANUAL_VALIDATION
S6 VALIDATED
S7 BUNDLING
S8 SEALED
S9 PROTOCOL_BREACH
Start
Terminal
Human required
Breach (halt)

Transitions are strictly enforced. Illegal state jumps (e.g., EXECUTING to SEALED without passing GATE_CHECK) trigger an automatic transition to PROTOCOL_BREACH, halting all operations until human recovery.

Dual-SHA commit tracking

Every sync captures two SHA256 hashes: the authority hash (content of authoritative files at sync time) and the repo hash (git HEAD at sync time). On bootstrap, both are verified. If either has diverged without a recorded transition, the system flags a potential integrity violation.

dual-SHA example
# Before sync
authority_sha: e4a1b2c3d4...
repo_sha: f5b2c3d4e5...
 
# After sync (both updated atomically)
authority_sha: 7c8d9e0f1a...
repo_sha: 8d9e0f1a2b...
Integrity verified. Chain unbroken.

Invariant registry

Named formal properties that must hold across all sessions. Each invariant is a human-readable assertion with an optional machine-checkable expression. If a session produces output violating a registered invariant, the gate check fails and the task cannot proceed to VALIDATED.

invariants.json (excerpt)
{
  "INV-001": {
    "name": "API response time under 200ms",
    "assertion": "p99_latency_ms < 200",
    "scope": "all endpoints",
    "established_by": "session-003-claude"
  },
  "INV-002": {
    "name": "No direct DOM manipulation in React components",
    "assertion": "grep -r 'document.getElementById' src/ == 0",
    "scope": "src/**/*.tsx",
    "established_by": "session-001-claude"
  }
}

Bundle generation pipeline (14 steps)

A deterministic handoff bundle is a self-contained ZIP with full project state, decision history, and SHA256 manifest. The generation pipeline runs in strict order:

1. Validate current state
2. Lock STATE_VECTOR
3. Snapshot decision log
4. Snapshot invariant registry
5. Capture file manifest
6. Hash all authoritative files
7. Record git HEAD
8. Serialize session history
9. Collect execution evidence
10. Compile gate results
11. Generate SHA256 manifest
12. Package into ZIP
13. Verify package integrity
14. Transition to SEALED

Protocol breach handling

DEL-v2 is fail-closed. Any of these conditions triggers an immediate transition to PROTOCOL_BREACH: illegal state transition attempt, SHA mismatch on bootstrap, invariant violation during gate check, unauthorized role action (e.g., Executor self-approving), or manual override without human confirmation. Recovery requires explicit human intervention - the system will not auto-recover.

Multi-agent coordination model

Role Responsibilities Constraints
Operator Defines tasks, sets invariants, approves transitions, performs manual validation Cannot execute implementation. Must be human or human-delegated.
Executor Implements code, runs tests, produces artifacts, reports gate results Cannot self-approve. Cannot skip gate checks. Cannot transition to SEALED.
Auditor Reviews execution evidence, verifies SHA integrity, validates invariants Cannot execute. Cannot modify state. Read-only access to all artifacts.
